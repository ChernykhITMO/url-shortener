# Coding Conventions

**Updated:** 2026-04-07

## Architecture, layers, and entry points
- The service remains a thin `net/http` stack with no third-party router so each layer is explicit (`README.md:1`, `router/router.go:1`). `cmd/app/main.go:1` sets up flags for storage selection (`inmemory` vs `postgres`), loads `config.InitConfig`, and wires CLI-level logging before handing the context to `cmd/app/http/app.go:1`, which instruments the server with structured `slog` logs, configuration-enforced timeouts, and graceful shutdown.
- Business logic lives in `internal/services` (validation, alias generation, retries) and depends on the `Storage` interface defined in `internal/services/service.go:11`. Storage implementations sit under `internal/storage`: `inmemory` uses a `sync.RWMutex`-guarded map (`internal/storage/inmemory/storage.go:1`), while `postgres` uses `database/sql` with `pgx` and exposes `New`/`Close` helpers around a `sql.DB` configured via `internal/config/config.go:1`.

## HTTP layer patterns
- Routes are wired with `http.NewServeMux` (no implicit middleware chains in third-party routers) and each endpoint is wrapped in `RequireMethodMiddleware`, `LoggingMiddleware`, and `RecoverMiddleware` before reaching a handler (`internal/http/router/router.go:1`, `internal/http/middleware/*.go`). Logging uses `log/slog` with contextual fields (`internal/http/middleware/logging.go:1`), and panics bubble through `RecoverMiddleware` so a 500 JSON error is emitted only once (`internal/http/middleware/recover.go:1`).
- Handlers decode and encode JSON with explicit limits: `CreateAlias` uses `http.MaxBytesReader`, `json.Decoder` with `DisallowUnknownFields`, `respond.WriteJSON`, and guarded `r.Body.Close()` (`internal/http/handlers/create_alias.go:1`). Errors returned from services are classified via `Handler.writeServiceError` by matching `services.Err*` sentinels (`internal/http/handlers/handler.go:1`). `GetURL` pulls aliases out of `r.PathValue` (Go 1.20+) and also funnels errors through the same responder before returning the DTO (`internal/http/handlers/get_url.go:1` and `internal/http/dto/response.go:1`).
- The DTO layer (request/response structs) keeps JSON tags localized in `internal/http/dto/request.go:1` and `internal/http/dto/response.go:1`, so handlers never marshal/unmarshal ad hoc maps. `respond.WriteJSON`/`WriteJSONError` centralize the `Content-Type` header, status, and encoding logic (`internal/http/respond/respond.go:1`).

## Business logic and domain rules
- URL validation/normalization happens inside `normalizeAndValidateURL` so callers just pass `raw URL -> cleaned URL` (see `internal/services/service.go:11`). Alias generation derives from `internal/domain/alias/alias.go:1` (length 10, alphabet `[a-zA-Z0-9_]`) and the service retries conflicts while still returning `services.ErrAttemptsExceeded` if retries are exhausted. Every exported error is a sentinel (`internal/services/errors.go:1`), and `wrapError` gives each function an `op` prefix so logs/stack traces can immediately show the failing layer.
- Storage adapters keep their SQL contracts explicit. `internal/storage/postgres/create_alias.go:1` uses a CTE with `INSERT ... ON CONFLICT ...` + `COALESCE` so duplicate original URLs consistently resolve to the same alias, `errors.As` against `pgconn.PgError` to convert constraint violations into `storage.ErrAliasConflict`/`storage.ErrInvalidAlias`, and a retry loop with context cancellation for lookups. Similar clarity exists in `internal/storage/postgres/get_alias.go:1` and the `selectAliasWithRetry` helper.

## Configuration & environment
- All YAML config lives under `internal/config/config.go:1`/`config/local.yaml:1`, which normalizes defaults, validates timeouts, and respects `HTTP_ADDR`/`POSTGRES_DSN` environment overrides. `config.Sections` (HTTP server, Postgres, service) map directly into structs used by the entrypoint and storage layers, so any config change follows the same pipeline.
- Logging is centralized through `slog` (`cmd/app/http/app.go:1`, `internal/http/handlers/handler.go:1`, `internal/http/middleware/logging.go:1`), and middleware always logs critical metadata (`method`, `path`, `status`, `duration`). Keep that `slog` + structured field pattern when adding instrumentation.

## Testing ethos (quality referent)
- Tests stub collaborators via small `MockStorage`/`MockService` structs (`internal/services/mock_storage_test.go:1`, `internal/http/handlers/mock_service_test.go:1`) so every unit test stays fast and deterministic. Handlers and middleware use `httptest.NewRequest`/`httptest.NewRecorder` plus `slog` loggers that write to `io.Discard` so they focus on behavior, not output capture.
- Table-driven tests dominate service behavior (`internal/services/create_alias_test.go:1`) and handler error handling (`internal/http/handlers/create_alias_test.go:1`, `internal/http/handlers/get_url_test.go:1`), often calling `t.Run` per scenario and checking both `status` codes and JSON bodies for precise responses. Follow those patterns: group expectations in structs, reuse helper mocks, and avoid silent failures by calling `t.Fatalf`/`t.Fatal` when expectations diverge.
